Purpose & Use Case:

Complete history of system actions for compliance, security, debugging, and accountability. Essential for: investigating "who changed what", meeting regulatory requirements (SOC 2, GDPR), troubleshooting user issues, detecting suspicious activity. Build when you need formal compliance or have security concerns.

Key Features Explained:
  • Action Types: Create, Update, Delete, Login, Export, Share, etc. Shows what happened.
  • User & Account Info: Who did it and which customer account. Critical for multi-tenant security.
  • Resource Type: What was affected (Appraisal, User, Account, Setting). Helps filter noise.
  • IP Address: Where action originated. Useful for detecting unauthorized access or debugging VPN issues.
  • Details Column: Specific information like "Changed status from Draft to Published". Provides context.
  • Date/Time: Precise timestamp. Essential for compliance and debugging.
  • Filters: By action, user, account, date range. Critical when logs have thousands of entries.
  • Export: Download logs for auditors or long-term storage.
Implementation Notes:

CAN SKIP UNTIL NEEDED. Requires: audit_logs table (user_id, account_id, action, resource_type, resource_id, details JSON, ip_address, created_at). Adds overhead to every database operation. Consider logging service (Loggly, Datadog). Start simple: just log critical actions (user creation, account changes, deletions). Expand coverage as compliance needs grow. Retention policy important (90 days? 1 year?).
